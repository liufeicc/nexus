# 对话历史自动记忆提取方案

## 背景

当前记忆系统完全依赖 LLM 主动调用 `memory_add` 工具写入，实际使用中几乎不会被触发（除非用户明确说"记住这个"）。需要在对话总结或清除时，后台自动提取有价值的信息写入记忆。

## 总体设计

在"总结历史对话"或"清除历史对话"时，异步启动后台 AGENT（使用副模型），分析完整对话历史，自主判断哪些信息值得记录，通过同样的记忆工具（memory_add/replace/search/remove）写入。

**核心思路**：
- 副模型扩展支持工具调用循环（tool_use → dispatch → tool_result → 再次 LLM）
- 记忆提取 AGENT 带着现有记忆作为上下文去分析，自主判断去重和更新
- 去重/更新逻辑靠 AGENT 语义判断：先 search，语义相同则跳过，内容有变化则 replace
- 异步不阻塞，失败静默

## 具体改动

### 1. `auxiliary-client.ts` — 副模型支持工具调用

新增 `callWithTools()` 方法，实现受控的工具调用循环：

```typescript
interface CallWithToolsParams {
  systemPrompt: string
  userPrompt: string
  tools: Array<{ name: string; description: string; parameters: object }>
  maxIterations?: number  // 默认 5
}

interface CallWithToolsResult {
  success: boolean
  iterationCount: number
}

async callWithTools(params: CallWithToolsParams): Promise<CallWithToolsResult>
```

核心逻辑：
- 循环 maxIterations 次
- 每次调用 `this.llm.chat(messages, { tools })` 传入工具 schema
- 无 toolCalls 则退出循环
- 有 toolCalls 则通过 `this.toolDispatch` 回调执行，结果追加到 messages
- 继续下一轮 LLM 调用

新增 `setToolDispatch(fn)` 设置工具分派回调。

**关键设计**：toolDispatch 作为回调由调用方注入，不改变 AuxiliaryClient 的纯 LLM 包装职责。

### 2. `memory-extractor.ts` — 新建记忆提取 AGENT

```typescript
class MemoryExtractorAgent {
  extract(messages: AgentMessage[]): Promise<void>
}
```

**核心流程**：
1. 检查消息数 < 4 则跳过（太短的对话不值得提取）
2. 创建/复用 AuxiliaryClient，注入 toolDispatch → SqliteMemoryProvider.handleToolCall
3. 构建 system prompt + user prompt
4. 调用 `auxClient.callWithTools()` 执行提取循环

**System Prompt**（完整内容）：

```
# 角色

你是一个记忆提取 Agent。你的任务是分析对话历史，提取值得长期保存的信息到记忆系统。

# 工作流程

1. 先用 memory_search 搜索现有记忆，查找与你要提取的信息相关的记录
2. 根据搜索结果判断：
   - 如果已有语义相同的记忆 → 跳过，不要重复记录
   - 如果主题相关但内容有更新/变化 → 用 memory_replace 更新旧记录
   - 如果无相关记忆 → 用 memory_add 新增
3. 重复以上步骤，直到所有值得记录的信息都已处理

# 应该记录什么

- 用户偏好和习惯（回复语言、代码风格、工作流程偏好等）
- 项目结构和关键文件路径（目录组织、模块职责、约定做法）
- 重要的技术决策及其原因（为什么选A不选B）
- 用户明确要求的规则和约束
- 反复出现的问题和对应的解决方案
- 环境配置（API key 位置、数据库连接、部署方式等）
- 对文件的重大操作（重构、重命名、迁移等高层级变更）

# 不应该记录什么

- 临时性的工具调用细节（读了一个文件内容、运行了一条命令）
- 中间过程的错误和重试（只记录最终解决方案）
- 寒暄和过渡语
- 已经记录过的重复信息
- 具体的代码改动（可从 git 历史查到）

# 记录格式

- 用中文记录（除非对话本身是英文）
- 每条记忆要简洁但完整，是一个独立可理解的陈述句
- 包含必要的上下文，如文件名、目录路径、技术名称
- 文件路径写完整
- 一条记忆只记录一个事实或决策

# scope 选择

- scope="user"：仅用户个人偏好和习惯
- scope="memory"：其他所有内容（技术决策、项目事实、文件操作等）

# 完成

完成所有提取后，输出一句简短总结即可。
```

### 4. `agent-service.ts` — 触发入口

**新增 `triggerMemoryExtraction(messages)` 函数**：

```typescript
function triggerMemoryExtraction(messages: AgentMessage[]): void {
  if (!agent || messages.length < 4) return

  const extractor = new MemoryExtractorAgent({
    nexusSessionId: agent.getNexusSessionId(),
    parentConfig: agent.getConfig(),
    summaryModelConfig: agent.getSummaryModelConfig(),
  })

  // Fire and forget，不阻塞
  extractor.extract(messages).catch(err =>
    logger.warn('[AIAgentManager] 记忆提取失败:', err)
  )
}
```

**修改 `compressConversationHistory()`**：
在请求压缩前，获取消息并触发提取：

```typescript
export function compressConversationHistory(sessionId: string = DEFAULT_SESSION_ID): boolean {

}
```

## 去重与更新机制

不通过代码逻辑实现，而是通过 **Prompt 指令 + memory_search 工具** 的组合：

1. Prompt 明确要求"每次新增/修改前，必须先用 memory_search 搜索相似记忆"
2. memory_search 使用 FTS5 全文搜索，返回语义相关的已有记忆
3. AGENT 根据搜索结果自主判断：
   - 已有相同语义 → 跳过
   - 主题相关但内容已变 → memory_replace 更新
   - 无相关信息 → memory_add 新增

## 异步不阻塞设计

```
触发点（清除/压缩）
    |
    +-- triggerMemoryExtraction()  -- 创建 MemoryExtractorAgent
    |       |
    |       +-- extractor.extract(messages)  -- Promise，不 await
    |              |
    |              +-- 副模型 LLM 循环 (max 8 iterations)
    |              +-- 调用 memory_search → 判断 → memory_add/replace
    |
    +-- 继续执行原有逻辑（clearHistory / requestCompression）
           （完全不等待提取完成）
```

## 错误处理

- **最外层**：`.catch()` 静默处理，只写日志，不影响清除/压缩操作
- **LLM 层**：单次 LLM 调用失败时，返回错误作为 tool result 给 LLM，让它自行调整
- **Tool 层**：SqliteMemoryProvider 已有完善的 try-catch
- **迭代限制**：maxIterations: 8 防止无限循环
- **SQLite 并发**：better-sqlite3 是单线程同步写入，不存在并发问题

## 验证方法

1. 手动对话后点击"压缩历史"或"清除历史"，查看日志中 `[MemoryExtractor]` 输出
2. 检查 `memory_entries` 表中是否有新增记录
3. 新会话启动时，验证提取的记忆是否能在对话中被检索到
4. 对同一主题反复对话两次并压缩，验证去重效果
