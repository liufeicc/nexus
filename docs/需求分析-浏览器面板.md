## 1. 需求概述

### 1.1 核心功能

在 Nexus 现有的终端面板基础上，新增一种"浏览器面板"类型。用户可以在一个面板内输入 URL 并加载网页，与终端面板并排展示、自由分割布局。

本质上是将 Nexus 从"纯终端多窗口管理器"扩展为"多类型面板工作台"。

### 1.2 用户场景与使用价值

| 场景 | 描述 | 价值 |
|------|------|------|
| 开发调试 | 左边终端运行开发服务器，右边浏览器实时预览页面 | 减少 Alt-Tab 切换，提升开发效率 |
| 日志监控 + 文档查阅 | 一边看服务器日志，一边查阅 API 文档 | 信息对照更直观 |
| Agent 控制网页（未来） | 通过 AI Agent 自动操作网页，如自动填表、爬取数据 | 自动化工作流的基础设施 |
| 多任务并行 | 终端、浏览器、未来更多面板类型自由组合 | 打造个性化工作台 |

### 1.3 设计原则

- **面板类型可扩展**：这次增加浏览器面板，未来还可能增加文件预览面板、图表面板等，架构上需要为扩展预留机制。
- **最小改动**：尽量复用现有的布局系统、快照系统和会话管理机制。
- **独立生命周期**：浏览器面板不依赖 PTY 进程，拥有独立的创建/销毁逻辑。

## 2. MVP（最小可用版本）功能清单

### 2.1 MVP 包含的功能

| # | 功能 | 描述 | 优先级 |
|---|------|------|--------|
| 1 | 面板类型区分机制 | 引入 `panelType` 字段（`'terminal' | 'browser'`），让系统能识别面板类型 | P0 |
| 2 | 创建浏览器面板 | 通过工具栏按钮或右键菜单创建浏览器面板 | P0 |
| 3 | URL 输入栏 | 浏览器面板顶部显示地址栏，可输入 URL 并回车加载 | P0 |
| 4 | 网页渲染 | 在面板内渲染加载的网页内容 | P0 |
| 5 | 前进/后退/刷新 | 基本浏览器导航按钮 | P0 |
| 6 | 页面标题显示 | 面板标题栏显示当前网页标题 | P0 |
| 7 | 与终端面板共存布局 | 浏览器面板可参与分屏、拖拽交换等布局操作 | P0 |
| 8 | 快照持久化 | 关闭后重新打开会话，浏览器面板能恢复上次的 URL | P0 |
| 9 | 关闭浏览器面板 | 关闭按钮正常关闭面板（无需杀死 PTY） | P0 |

### 2.2 MVP 不包含的功能（后续迭代）

| 功能 | 排除原因 |
|------|----------|
| Agent 控制网页 | 属于远期规划，需要 AI 集成和自动化框架，不在 MVP 范围 |
| 开发者工具（DevTools） | 开发调试场景有用，但 MVP 可以先不做 |
| 书签/历史记录 | 浏览器面板是嵌入工具，不是完整浏览器，这些功能优先级低 |
| Cookie/LocalStorage 隔离 | 每个面板共享同一 Electron 环境即可，后续有需求再做隔离 |
| 下载管理 | MVP 阶段不需要，网页内的下载走 Electron 默认行为 |
| 多标签页（浏览器内） | 面板本身就是标签概念，不需要在面板内再嵌套标签 |

## 3. 技术可行性分析

### 3.1 Electron 嵌入浏览器的三种方案对比

| 方案 | 实现方式 | 优点 | 缺点 | 适用性 |
|------|----------|------|------|--------|
| **方案A: `<webview>` 标签** | 使用 Electron 内置的 `<webview>` 标签 | 1. 独立进程运行，不影响主渲染进程性能<br>2. 内置导航 API（goBack, goForward, reload, loadURL）<br>3. 有独立的 `webContents`，可注入脚本<br>4. 天然隔离，网页崩溃不影响 Nexus UI<br>5. 支持 `preload` 脚本注入（为未来 Agent 控制预留） | 1. Electron 官方不推荐（建议用 BrowserView）<br>2. 在 React 中使用需要特殊处理（`webview` 是自定义元素）<br>3. 安全性需通过 `webview` 属性严格控制 | **推荐** |
| **方案B: `<iframe>`** | 标准 HTML iframe 元素 | 1. 实现最简单，纯前端<br>2. React 中直接使用 | 1. **严重安全限制**：很多网站通过 `X-Frame-Options` 或 CSP 拒绝被 iframe 嵌入（如 Google、GitHub）<br>2. 无法获取页面标题、URL 变化等信息<br>3. 无法前进/后退<br>4. 跨域限制导致无法注入脚本 | **不推荐** |
| **方案C: BrowserView / BrowserWindow** | Electron 主进程创建 BrowserView，嵌入到渲染进程的指定区域 | 1. 官方推荐方案<br>2. 完整浏览器能力<br>3. 独立进程 | 1. **需要 IPC 通信管理**：渲染进程需要告诉主进程"在哪个坐标显示 BrowserView"<br>2. 布局变化（分屏、拖拽、resize）时需要实时同步 BrowserView 的位置和尺寸<br>3. 实现复杂度高，容易出 bug<br>4. 与当前架构冲突大 | 适合完整浏览器应用，不适合嵌入式面板 |

### 3.2 推荐方案：`<webview>` 标签

**理由：**

1. **改动最小**：`<webview>` 在渲染进程中使用，不需要修改主进程代码和 IPC 通道（除了新增一个 `browser:create` 之类的可选通道）。
2. **布局兼容性好**：作为 React 组件内的 DOM 元素，自动跟随 flex 布局调整尺寸，不需要手动同步位置。
3. **功能完整**：内置导航 API，前进/后退/刷新一句话调用。
4. **未来可扩展**：通过 `preload` 脚本注入，为未来 Agent 控制网页提供通道（Agent 可以通过 IPC 向 `webview` 的 `webContents` 注入脚本）。
5. **性能隔离**：网页在独立进程中运行，即使加载复杂页面也不会阻塞 Nexus 的 UI 渲染。

**注意事项：**
- `<webview>` 需要在 `webPreferences` 中启用 `webviewTag: true`
- React 中使用时，需要用 `useEffect` 获取 webview 元素并绑定事件
- 需要设置 `webpreferences` 限制权限，防止安全风险

### 4.3 状态管理扩展

#### 4.3.1 新增方法

| 方法 | 描述 | 与终端面板的差异 |
|------|------|-----------------|
| `createBrowserPanel(url?: string)` | 创建浏览器面板 | **不调用 PTY**，直接构建 PanelState 添加到 store |
| `updateBrowserUrl(panelId, url)` | 更新浏览器面板 URL | 终端面板没有对应操作 |

#### 4.3.2 需要修改的方法

| 方法 | 修改内容 |
|------|----------|
| `closePanel(panelId)` | 判断 `panelType`，终端面板调用 `pty.kill()`，浏览器面板直接跳过 PTY 终止 |
| `restorePanelsFromData(...)` | 根据 `panelType` 分别处理：终端面板重建 PTY，浏览器面板仅恢复 URL |
| `saveSnapshot(...)` | 快照数据增加 `panelType` 和 `url` 字段 |

#### 4.3.3 兼容策略

- `ptyId` 从必填改为可选（`ptyId?: string`），现有终端面板逻辑不变
- `cwd` 从必填改为可选（`cwd?: string`），现有终端面板逻辑不变
- 所有现有代码中引用 `ptyId` 和 `cwd` 的地方需要加可选链 `?.` 或空值检查

### 4.4 生命周期变化

#### 终端面板生命周期（不变）
```
创建: create PTY → addPanel → 初始化布局 → 保存快照
关闭: kill PTY → removePanel → 简化布局 → 保存快照
恢复: 重建 PTY → restorePanelsFromData
```

#### 浏览器面板生命周期（新增）
```
创建: 生成 panelId → addPanel (无 PTY) → 初始化布局 → 保存快照
关闭: removePanel (无 PTY 终止) → 简化布局 → 保存快照
恢复: 读取 url → 恢复面板状态 (无 PTY 重建)
```

### 4.5 快照系统兼容

- **向后兼容**：现有快照数据中没有 `panelType` 字段，读取时默认为 `'terminal'`
- **数据库表结构**：当前快照以 JSON 格式存储在 SQLite 中（`layoutData` 和 `panelStates` 字段都是 JSON），**无需修改表结构**，只需在 JSON 中增加字段
- **恢复逻辑**：恢复快照时，根据 `panelType` 决定是否需要重建 PTY

## 5. 未来扩展规划

### 5.1 面板类型扩展机制

建议引入"面板类型注册表"模式，为未来增加更多面板类型预留扩展点：

```typescript
// src/core/types/panel-registry.ts（建议新增）

// 面板类型枚举，新增类型只需在此添加
export type PanelType = 'terminal' | 'browser' | 'file-preview' | 'chart' | ...

// 面板类型元数据
export interface PanelTypeMeta {
  type: PanelType
  label: string          // 显示名称，如 "终端"、"浏览器"
  icon: string           // 图标
  requiresPty: boolean   // 是否需要 PTY
  defaultTitle: string   // 默认标题模板
  snapshotFields: string[] // 快照时需要保存的字段
}

// 注册表
export const PANEL_TYPES: Record<PanelType, PanelTypeMeta> = {
  terminal: {
    type: 'terminal',
    label: '终端',
    icon: 'terminal',
    requiresPty: true,
    defaultTitle: '终端 - {cwd}',
    snapshotFields: ['ptyId', 'cwd', 'title'],
  },
  browser: {
    type: 'browser',
    label: '浏览器',
    icon: 'globe',
    requiresPty: false,
    defaultTitle: '{title}',
    snapshotFields: ['url', 'title'],
  },
}
```

**好处：**
- 新增面板类型时，只需注册元数据，不需要到处改 `if-else`
- 布局渲染器可以根据注册表动态决定渲染哪个组件
- 快照系统可以根据 `snapshotFields` 自动序列化/反序列化

### 5.2 Agent 控制网页的前置需求

要实现"Agent 自动控制网页行为"，需要以下基础设施：

| 前置需求 | 描述 | 优先级 |
|----------|------|--------|
| webview preload 脚本注入 | 通过 `<webview>` 的 `preload` 属性注入脚本，建立与主进程的通信通道 | P1 |
| webview webContents 访问 | 主进程通过 `remote` 或 IPC 获取 webview 的 `webContents`，用于执行脚本和获取页面信息 | P1 |
| DOM 操作 API | 提供一组 API 让 Agent 可以：获取页面元素、点击、输入、获取文本等 | P2 |
| 页面状态报告 | webview 向主进程报告：页面加载状态、DOM 变化、网络请求等 | P2 |
| Agent 指令通道 | 新增 IPC 通道 `agent:execute`、`agent:result` 等 | P2 |

**技术路径：**
1. MVP 阶段：`<webview>` + `preload` 脚本，建立基础通信
2. 第二阶段：实现 DOM 操作 API 封装
3. 第三阶段：与 AI Agent 集成，通过自然语言指令控制网页

### 5.3 与 AI 功能的集成点

| 集成点 | 描述 |
|--------|------|
| 智能 URL 推荐 | 根据终端命令输出中的 URL，AI 自动推荐在浏览器面板中打开 |
| 终端-浏览器联动 | 在终端启动开发服务器后，AI 自动在浏览器面板中打开对应地址 |
| 页面内容分析 | AI 分析浏览器面板中的页面内容，与终端操作联动 |
| 自动化脚本生成 | 用户描述需求，AI 生成操作网页的脚本并执行 |

## 6. 风险和注意事项

### 6.1 安全风险

| 风险 | 描述 | 应对措施 |
|------|------|----------|
| XSS 攻击 | 恶意网页通过 `<webview>` 执行脚本攻击 Nexus | 1. 设置 `webview` 的 `nodeintegration="false"`<br>2. 设置 `contextIsolation="true"`<br>3. 不使用 `allowpopups` 除非必要 |
| 跨站请求伪造 | 恶意页面通过 `<webview>` 访问本地资源 | 1. 限制 `webview` 的权限<br>2. 不暴露敏感 IPC 通道 |
| 导航劫持 | 恶意页面将 `<webview>` 导航到钓鱼页面 | 1. 监听 `will-navigate` 事件<br>2. 地址栏始终显示真实 URL |
| 第三方 Cookie | 网页可能设置 Cookie 追踪用户 | 后续可考虑使用 `partition` 属性隔离不同面板的存储 |

### 6.2 性能考虑

| 考虑点 | 描述 | 应对措施 |
|--------|------|----------|
| 多个 webview 同时加载 | 用户可能同时打开多个浏览器面板 | 1. 非激活面板可以暂停渲染（类似浏览器标签页休眠）<br>2. 设置合理的内存上限 |
| webview 初始化开销 | `<webview>` 启动有几十毫秒到几百毫秒延迟 | 1. 创建时先显示 loading 占位<br>2. 考虑懒加载 |
| 大页面内存占用 | 复杂网页（如 IDE、地图）内存占用大 | 1. 面板最小尺寸限制<br>2. 提供"重新加载"按钮清理内存 |

### 6.3 兼容性

| 兼容性 | 描述 | 应对措施 |
|--------|------|----------|
| X-Frame-Options | 部分网站拒绝被嵌入 | `<webview>` 不受此限制（独立进程），但部分网站有 CSP 限制 |
| 登录态共享 | `<webview>` 有独立的 Cookie 存储，不会共享主窗口登录态 | 这是优点（安全隔离），但用户可能需要在浏览器面板内重新登录 |
| Linux 沙箱问题 | 当前 Electron 使用 `--no-sandbox` 启动 | `<webview>` 在 `--no-sandbox` 下工作正常 |
| Electron 版本兼容性 | 不同 Electron 版本对 `<webview>` 支持不同 | 当前使用 Electron 30，`<webview>` API 稳定 |

### 6.4 开发注意事项

| 注意点 | 描述 |
|--------|------|
| React 中使用 webview | `<webview>` 是自定义元素，React 不能直接绑定事件，需要用 `useEffect` + `addEventListener` |
| TypeScript 类型 | `<webview>` 的类型定义在 `@types/electron` 中，需要引入 |
| 样式隔离 | `<webview>` 有独立的 shadow DOM，Nexus 的 CSS 不会影响其内容 |
| 调试支持 | 可以在主进程中调用 `webview.getWebContents().openDevTools()` 打开开发者工具 |
