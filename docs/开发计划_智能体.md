# 智能体开发计划：将 Hermes Agent 核心循环移植到 Nexus

## Context

Nexus 是一个 Electron + React + TypeScript 的终端应用。Hermes Agent 是一个 Python 编写的全功能 AI 智能体框架。

**目标**：用 TypeScript 重写 Hermes Agent 的核心智能体循环，集成到 Nexus 项目中。
- 灵动岛（Dynamic Island）作为智能体的输入/输出界面
- 不需要子智能体委托
- 不需要独立 UI 组件

## Hermes 核心循环（我们要移植的东西）

```
while 预算未耗尽:
  1. 构建系统提示（身份 + 工具列表 + 记忆）
  2. 调用 LLM API
  3. 如果返回 tool_calls → 执行工具 → 结果追加到消息历史 → 继续循环
  4. 如果返回纯文本 → 结束循环，返回给用户
  5. 检查上下文是否接近上限 → 必要时压缩
```

## 实施顺序

### 第一阶段：基础设施（必须先做，其他步骤依赖这些）

#### 1. 定义核心类型 — `src/core/types/agent.ts`

**作用**：定义整个智能体系统的数据结构，确保所有模块对"一条消息"、"一个工具"、"一次调用结果"有统一的理解。

**为什么先做**：TypeScript 是强类型语言，所有后续模块都依赖这些类型。没有它们，编译器和 IDE 都无法提供类型检查和自动补全。

**包含内容**：
- `AgentConfig` — 智能体配置（模型名、API 地址、最大迭代次数、API Key 等）
- `AgentMessage` — 一条对话消息（角色是 user/assistant/system/tool，内容，可能包含工具调用）
- `ToolDefinition` — 工具定义（工具名如 "read_file"、描述、参数 JSON Schema、处理函数）
- `ToolResult` — 工具执行结果（成功还是失败、输出文本）
- `AgentState` — 智能体状态（空闲/运行中/已停止/出错）
- `IterationBudget` — 迭代预算（防止智能体无限循环，默认最多 90 次工具调用）

#### 2. LLM 客户端 — `src/main/agent/llm-client.ts`

**作用**：统一与各种 LLM API 的通信。屏蔽 OpenAI、Anthropic 等不同 API 的差异。

**为什么第二个做**：智能体的核心能力来自 LLM，没有它其他模块无法验证。这是最独立的模块之一，不依赖其他步骤。

**核心能力**：
- 统一的消息发送接口（支持 OpenAI Chat Completions API 和 Anthropic Messages API）
- 流式响应支持（SSE，逐字输出）
- 自动重试（超时、限流时指数退避）
- 错误分类（是限流还是认证失败还是上下文超长）

### 第二阶段：工具系统（智能体的"手脚"）

#### 3. 工具注册系统 — `src/main/agent/tool-registry.ts`

**作用**：统一管理所有工具的注册、查询和调用。类似"工具黄页"。

**核心方法**：
- `register(name, schema, handler)` — 注册一个工具
- `getDefinitions()` — 获取所有工具的 OpenAI 格式 schema（传给 LLM API 用）
- `dispatch(name, args)` — 根据名字调用工具

#### 4. 内置工具实现 — `src/main/agent/tools/*.ts`

**作用**：实现智能体可以调用的具体能力。

**首批工具及作用**：

| 工具 | 文件 | 作用 |
|------|------|------|
| `terminal` | `terminal-tool.ts` | 执行终端命令。复用 Nexus 现有的 PtyService |
| `read_file` | `file-tools.ts` | 读取文件内容 |
| `write_file` | `file-tools.ts` | 写入/修改文件 |
| `search_files` | `file-tools.ts` | 搜索文件内容（类似 grep）|
| `web_search` | `web-tools.ts` | 网络搜索 |
| `todo` | `todo-tool.ts` | 任务列表，智能体用它规划多步骤任务 |

### 第三阶段：核心智能体（组装所有零件）

#### 5. 提示构建器 — `src/main/agent/prompt-builder.ts`

**作用**：组装发给 LLM 的系统提示词。告诉 AI 它是谁、能做什么、该怎么工作。

**分层组装**：
```
┌─────────────────────────────────┐
│ 身份层: "你是一个AI助手..."      │
│ 工具指南: "你必须使用工具..."    │
│ 可用工具列表: read_file, ...     │
│ 记忆: "之前你提到..."            │
│ 上下文文件: 项目规则文件内容      │
└─────────────────────────────────┘
```

#### 6. 核心智能体类 — `src/main/agent/ai-agent.ts`

**作用**：整个智能体的"大脑"。实现核心对话循环。

**核心方法 `runConversation` 的执行流程**：
```
用户输入 → 构建系统提示 → 调用 LLM → 判断返回类型
                                              │
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
                              tool_calls              纯文本
                                    │                   │
                                    ▼                   │
                            执行工具 → 结果加入历史      │
                                    │                   │
                                    └─────┬─────────────┘
                                          ▼
                                    回到调用 LLM 继续循环
                                    （直到返回纯文本或预算耗尽）
```

**关键机制**：
- 迭代预算：最多 90 次循环
- 并行工具执行：多个只读工具可并行
- 中断机制：用户可随时中断
- 回调通知：每步进展通知灵动岛

### 第四阶段：上下文管理

#### 7. 上下文压缩器 — `src/main/agent/context-compressor.ts`

**作用**：当对话历史太长、接近模型上下文窗口上限时，自动压缩中间对话。

**为什么放在后面**：前几轮对话不太可能撑爆上下文窗口，压缩器是"锦上添花"的优化，核心循环可以先跑起来再优化。

### 第五阶段：通信集成

#### 8. IPC 接口 — 修改现有文件

**作用**：让渲染进程（前端 React）能与主进程的智能体通信。

**修改的文件**：

| 文件 | 修改内容 |
|------|---------|
| `src/core/constants/ipc-channels.ts` | 新增频道名：`agent:run`、`agent:stop`、`agent:stream`、`agent:status` |
| `src/main/ipc/ipc-handlers.ts` | 注册 IPC 处理器 |
| `src/main/preload.ts` | 暴露 `window.electron.agent.run()` 等 API |

**通信流程**：
```
灵动岛输入 → preload.ts → ipc-handlers.ts → AIAgent.runConversation()
AIAgent 回调 → ipc-handlers.ts → webContents.send() → 灵动岛显示输出
```

### 第六阶段：用户界面

#### 9. 灵动岛集成 — 修改 `src/main/services/dynamic-island.ts`

**作用**：将灵动岛作为智能体的输入输出界面。

**具体改动**：
- 灵动岛输入框的消息同时发送给智能体 IPC
- 智能体的流式输出显示在灵动岛输出区域
- 工具执行状态在灵动岛实时显示
- 提供"停止"按钮

## 开发顺序总览

下面是所有文件的线性开发顺序，每一步都是下一步的前提：

```
 1. src/core/types/agent.ts          ← 类型定义（所有模块依赖）
    │
    ▼
 2. src/main/agent/llm-client.ts     ← LLM 客户端（最独立，先验证 API）
    │
    ▼
 3. src/main/agent/tool-registry.ts  ← 工具注册中心
    │
    ▼
 4. src/main/agent/tools/*.ts        ← 具体工具实现（依赖 registry + types）
    │
    ▼
 5. src/main/agent/prompt-builder.ts ← 提示构建（依赖 types）
    │
    ▼
 6. src/main/agent/ai-agent.ts       ← 核心智能体（组装前面所有模块）
    │
    ▼
 7. src/main/agent/context-compressor.ts ← 上下文压缩（优化项，可延后）
    │
    ▼
 8. src/core/constants/ipc-channels.ts   ← 新增 IPC 频道名
    │
    ▼
 9. src/main/ipc/ipc-handlers.ts     ← IPC 处理器，连接 ai-agent.ts
    │
    ▼
10. src/main/preload.ts             ← 暴露 API 给渲染进程
    │
    ▼
11. src/main/services/dynamic-island.ts ← 灵动岛集成（最后接 UI）
```

**简化版口诀**：先类型 → 再 API → 再工具 → 组装智能体 → 接 IPC → 最后接 UI

## 验证方案

1. **阶段 1-2 验证**：测试 LLM 客户端能否成功调用 API 并返回结果
2. **阶段 3-4 验证**：测试工具注册和调用循环，发送 "列出当前目录文件"，验证能否正确执行 terminal 工具
3. **阶段 5-6 验证**：测试 IPC 通信和灵动岛集成，端到端验证完整流程
