## 四、失败用例分析

| 编号 | 文件 | 用例 | 原因分类 | 详细说明 |
|------|------|------|----------|----------|
| 1 | session.dao | 列表按创建时间降序排列 | 测试设计问题 | 两条记录在同一秒内插入，created_at 字段精度为秒级，导致排序结果不确定 |
| 2 | snapshot.dao | 保存快照并返回 ID | MOCK 配置问题 | crypto.randomUUID 的 mock 未生效，DAO 内部使用真实的 randomUUID 生成了 UUID |
| 3 | browser-history.dao | 保留最近 500 条记录 | 对源码理解偏差 | DAO 的 save() 方法未实现 500 条自动清理逻辑，自动清理仅在 list() 的 LIMIT 参数中 |
| 4 | browser-history.dao | 超过 500 条自动删除最旧记录 | 对源码理解偏差 | 同上 |
| 5 | browser-history.dao | 按 ID 删除历史记录 | 对源码理解偏差 | DAO 的 delete() 方法参数是 url 而非 id，测试传入了错误类型的参数 |
| 6 | browser-history.dao | 不影响其他记录 | 对源码理解偏差 | 同上，delete() 参数类型不匹配 |
| 7 | constants | applyTheme 应用主题到文档 | 运行环境问题 | vitest 默认运行在 node 环境，不存在浏览器 document 对象 |
| 8 | constants | getBasename 尾斜杠处理 | 对源码理解偏差 | 源码对尾部斜杠路径不做截断处理，测试期望值与实际行为不符 |

**结论**: 8 个失败用例均为测试用例本身的问题（MOCK 不准确、对源码行为理解有偏差、运行环境差异），**未发现任何 DAO 源码 bug**。

## 五、修复建议

| 失败用例 | 修复方式 |
|----------|----------|
| SessionDAO 列表排序 | 手动为两条记录设置不同的 created_at 时间戳（间隔 1 秒以上） |
| SnapshotDAO 保存返回 ID | 去掉对 mock ID 的断言，改为验证返回值非空且为字符串类型 |
| BrowserHistory 500 条限制 | 确认 DAO 源码是否在 save 时实现了自动清理，若未实现则移除该用例 |
| BrowserHistory delete 参数 | 将 delete 参数改为传 url 而非 id，与 DAO 实际签名匹配 |
| applyTheme 应用主题 | 将 vitest 环境设置为 jsdom，或移除该用例（属于浏览器端功能） |
| getBasename 尾斜杠 | 修改期望值为实际返回值，或移除该用例 |

### MOCK 策略

| 组件 | MOCK 方式 |
|------|-----------|
| 数据库 | better-sqlite3 内存模式（:memory:），完整 schema 在 setup.ts 中创建 |
